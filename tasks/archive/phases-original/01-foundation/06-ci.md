# 01.06 CI pipelines

## Context

CI runs on every PR. Tests, linting, type checks, and OpenAPI drift detection across all three apps.

## Goal

Working GitHub Actions workflows that gate merges.

## Workflows

### `.github/workflows/api.yml`

Triggers on push to `main` and PRs touching `apps/api/**`, `.github/workflows/api.yml`.

Jobs:
- `lint`: ruff check, ruff format --check.
- `typecheck`: mypy or pyright strict on `apps/api/app`.
- `test`: spin up Postgres + Redis services, run `pytest -q`.
- `openapi`: start the app, fetch `/openapi.json`, write to `packages/openapi/openapi.json`. Fail if `git diff` shows changes (forces the dev to commit the regenerated spec).

### `.github/workflows/web.yml`

Triggers on `apps/web/**`.

Jobs:
- `lint`: next lint, prettier --check.
- `typecheck`: tsc --noEmit.
- `build`: next build with mock env values.
- `openapi-sync`: run `openapi-typescript` against `packages/openapi/openapi.json`, fail on diff (keeps generated types committed and current).

### `.github/workflows/ios.yml`

Triggers on `apps/ios/**`.

Jobs:
- `build`: xcodebuild on macos-14 runner, scheme `GymApp`, destination iOS Simulator iPhone 15.
- `test`: xcodebuild test, same destination.
- `openapi-sync`: regenerate Swift OpenAPI client and fail on diff.

### `.github/workflows/all.yml`

Single matrix workflow that runs all three on the `main` branch nightly to catch flakes.

## Deliverables

1. All four workflow files.
2. README section under `docs/ci.md` explaining what each job does and how to debug a red build.
3. Branch protection rules documented in `docs/ci.md` (require all three to pass before merge).

## Acceptance criteria

- A trivial PR that touches only the README does not run app workflows.
- A PR that changes the API schema fails the openapi-sync job in web and iOS until both are regenerated.
- iOS workflow caches DerivedData and SwiftPM so subsequent runs complete in under 10 minutes.

## Dependencies

- `01.02 FastAPI skeleton`
- `01.05 Next.js web skeleton`
- `08.01 iOS app skeleton` (the iOS workflow can be added after that task, or stubbed first)

## Out of scope

- Production deploy workflows (those live under `09-deployment`).
